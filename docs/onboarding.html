Der Loop bleibt agentisch. Verändert ist allein, woran sein Haltepunkt hängt: an einem externen, getypten Prädikat statt an generator-interner Zufriedenheit.

SPOT — die getypte Referenz

SPOT (Single Point of Truth): ein getypter F#-DU-Graph. Ein JSON pro Knoten unter .spot/ — git-diffbar, serverlos.

Jeder Knoten trägt einen Konvergenz-Status und Relationen. Alles andere (Plan, Dev, Infra, Doku) ist eine Projektion dieses einen Modells.

Knotenarten: Spec · Test · Term · Decision · Component · Invariant · Premise · Risk · Knowledge · Tool
Relationen: RelatesTo · DependsOn · SpecRef · Supersedes


// Cdd.Core/Spot.fs — verbatim
type Convergence =
    | Diverged   // Code weicht vom Modell ab
    | Aligned    // Modell und Code synchron
    | Pending    // Modell da, Code noch nicht
    | Orphaned   // Code da, kein Modell

Die vier Konvergenz-Zustände

stateDiagram-v2 [*] --> Pending: Modell da, Code noch nicht Pending --> Aligned: echter grüner Test deckt Knoten Aligned --> Diverged: Code weicht vom Modell ab Diverged --> Aligned: erneut gegen Spec konvergiert [*] --> Orphaned: Code ohne Modell Orphaned --> Pending: Modell nachgezogen

Konvergenz ist kein Fixpunkt-Versprechen, sondern ein beobachtbarer Zustand pro Knoten. Der Loop terminiert, wenn kein Knoten mehr Diverged oder Orphaned ist.

Das Orakel: echter Test, nicht Exit-0

sequenceDiagram autonumber participant Gen as Generator (LLM) participant Code as Code-Pfad participant Or as Orakel (Gate) participant Test as dotnet test participant Spot as .spot/ Knoten Gen->>Code: Vorschlag (kein Schreibrecht aufs Gate) Or->>Test: führt echten Lauf aus Test-->>Or: Ergebnis alt Test-PASS und Marker deckt Knoten Or->>Spot: Zustand := Aligned else Exit-0 ohne PASS / leer / übersprungen Or->>Spot: bleibt Pending / Diverged end

Maßgeblich ist das Test-PASS, nicht der Prozess-Exitcode 0. Ein Prozess kann mit Exit-0 enden, ohne dass eine Property geprüft wurde.

Zwei Fallstudien

Ein Existenznachweis — und sein hartes Gegenstück.

Warum zwei?

flowchart LR subgraph A["🎮 Runenruf — Existenznachweis"] direction TB a1["Simulationsdomäne · ~125 LoC"] a2["Spec = Welt (freundlichste Domäne)"] a3["46/46 · FsCheck über jeden Seed"] end subgraph B["📒 Ledger — Gegenstück"] direction TB b1["echtes IO: Journal + Replay · ~41 LoC Kern"] b2["Spec ≠ Welt · veränderliches Requirement"] b3["6/6 · + 2 Lean-Beweise"] end A ~~~ B style A fill:#0f1b14,stroke:#3fb950 style B fill:#13243a,stroke:#4c9aff

Runenruf zeigt, dass die Maschinerie geschlossen durchläuft. Ledger zeigt, was passiert, wenn die Spec die Welt nicht automatisch trifft.

🎮 Fallstudie A: Runenruf

  • 46/46 Tests (autoritativer Runner, kein grep) @ 56376a5
  • Tragende FsCheck-Property (property-based testing) spec-siegel-lager-nichtnegativ: quantifiziert über jeden Seed und jede Operationsfolge — Nichtnegativität des Lagerbestands
  • Deterministisch: gleicher Seed → gleicher Lauf → Gegenbeispiel reproduzierbar

Ehrlich: die freundlichste Domäne, die wir kennen — die Spezifikation ist die Welt. Das belegt Anwendbarkeit und Mechanik, nicht den harten Realismus-Fall. A 46/46 · B „trifft die Welt" entfällt hier.

📒 Fallstudie B: Ledger

Doppisches Hauptbuch. Echtes IO: persistentes Journal + Replay-Pfad. Veränderliches Requirement (Gebühren).

  • F#/.NET 9, Beträge als int64-Cent
  • ~41 LoC Domänenkern
  • 6/6 Tests @ 0f39ab0

Hier ist „trifft das Modell die Welt" nicht leer — und genau hier zeigt sich, wo das Loop-interne Orakel endet.

Mis-Spec = eine grüne, aber falsche Spezifikation. → nächste Folie: wie er gefangen wurde.

Der gefangene Mis-Spec — ein unabhängiges Gegen-Orakel

stateDiagram-v2 [*] --> Gruen1 Gruen1: Korrektes Modell · 6/6 grün Gruen1 --> Falsifiable: naives Modell injiziert (Gebühr verschwindet) Falsifiable: rot — nach 4 Tests Falsifiable --> Gruen2: revert + Umstellung auf Doppik Gruen2: Wert bleibt neutral · 6/6 grün Gruen2 --> [*] note right of Falsifiable Nicht der Loop fand den Fehler, sondern eine vom Generator UNABHÄNGIGE Quelle (hier: der Mensch): stärkere Property + geändertes Requirement als Dekorrelator. Ein zweiter unabhängiger Agent könnte dieselbe Rolle einnehmen — eine Setzung, kein Unmöglichkeitsbeweis. end note

Erste, schwächere Properties akzeptierten eine fehlerhafte Buchungslogik — verifizierer-akzeptiert ≠ korrekt (VeriAct). Erst die externe Setzung machte die grüne Spec falsifizierbar.

Das stärkste Bild: das Gate beim Scheitern


$ ./demo-gate-at-failure.sh            # @ 0f39ab0 — ein NEGATIV-Artefakt
  1) korrektes Modell            6/6  grün
  2) naives Modell injiziert     (Gebühr verschwindet)
  3) Orakel:  Falsifiable        rot   ← verwirft das falsche Modell
  4) revert                      6/6  grün

Ein grüner Lauf ist ein Cursor-Klon mit Vokabular. Ein reproduzierbar erzwungenes rotes Resultat ist der reproduzierbare Existenznachweis, dass das Gate diskriminiert.

Der Lean-Beweis: Werterhaltung (Lean 4 — Theorembeweiser)


-- proofs/Werterhaltung.lean @ 0f39ab0 (verbatim)
theorem werterhaltung (hb : Hauptbuch) (bs : List Buchung) :
    total (buchen hb bs) = total hb := by
  unfold buchen
  induction bs generalizing hb with
  | nil => rfl
  | cons b bs ih =>
      simp only [List.foldl_cons]
      rw [ih (buche hb b), werterhaltung_einzel]
  • sorry-frei — #print axioms werterhaltung → nur propext, Quot.sound (keine Zusatz-Axiome)
  • ohne Mathlib (nur Core + omega) · CI-Job lean-proof grün
  • Das Lean-Modell bildet den Deckungs-/Ablehnungspfad der F#-Funktion buche nach (manuell aligned, nicht maschinell extrahiert) → kein vakuum-starker Beweis

Teil 6

Ehrlichkeit & Stand

Mit einer Lücke argumentieren, nicht mit einem Versprechen.

Ehrliche Grenzen

  • N = 2 Fallstudien — Anwendbarkeit, nicht Skalierung; beide klein
  • Zwei Lean-bewiesene Invarianten (Werterhaltung + Nichtnegativität); die übrigen (Isolation, Replay) nur FsCheck (Roadmap)
  • Verifizierer-akzeptiert ≠ korrekt (VeriAct) — das Orakel allein genügt nicht
  • Preprint, nicht peer-reviewed

Anti-Lucas (wichtig): Das Setzen der Grund-Invariante ist im System nicht selbst-fundierbar — ein Regress (architektonisches Argument, kein Unmöglichkeitsbeweis). Dass der Mensch den Anker setzt, ist eine Wahl (Intent ist extern + verantwortungsbehaftet), nicht „unmechanisierbar". Ein zweiter, unabhängiger Agent könnte ihn setzen. Penrose/Lucas wird abgelehnt.

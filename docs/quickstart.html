● Quick Guide · ~10 Minuten · kein Setup

CDD selbst sehen — im Browser

Ohne Installation: Umgebung im Browser starten, Tests grün sehen, und beobachten, wie das Gate einen gefälschten grünen Test abweist.

Schritt 1 · ~1 Min

Im Browser öffnen (Codespaces)

Ein Klick startet die echte Umgebung in der Cloud — nichts lokal zu installieren (GitHub-Login genügt):

☁️ Open in Codespaces

Alternativ lokal: git clone https://github.com/Koschnag/cong-driven-development (braucht .NET 9).

Schritt 2 · ~2 Min

Alles grün? Die Suite laufen lassen

dotnet test tests/Cdd.Tests
# erwartet: Passed! 42/42

Darunter die Tests des harten Gates selbst — inklusive zweier FsCheck-Properties über zufällige Eingaben.

Schritt 3 · ~3 Min — der Aha-Moment

Das Gate weist Fake-Grün ab

Öffne src/Cdd.Core/Gate.fs und den Test tests/Cdd.Tests/Tests.fs (suche „failwith-TODO-Skelett"). Der Kern in Worten:

# Ein Test-Skelett trägt den Marker, ist aber „failwith TODO" → Lauf ROT.
# Altes Marker-Orakel:  Sync.SetzeSpecAligned → Aligned  (fälschlich „fertig")
# Hartes Gate (rot):     Gate.setzeAlignedWennGruen → Pending  (blockiert)
# Hartes Gate (grün):    Gate.setzeAlignedWennGruen → Aligned  (echt fertig)

Genau das ist der Punkt: „fertig" ist gemessen, nicht behauptet — und die KI kann sich kein Grün erschleichen.

Schritt 4 · ~4 Min

Den Spec → Test → Code-Fluss selbst fahren

cdd init                     # getyptes SPOT-Modell (Beispiel-Spec) anlegen
cdd derive-tests --write     # aus jedem Akzeptanzkriterium ein Test-Knoten
cdd derive-code --out tests/Derived.fs   # Test-Skelett (failwith → rot)
# … Implementierung + echten Test schreiben …
cdd sync-tests --write       # misst Konvergenz: Aligned erst bei echtem Grün
cdd diff                     # Konvergenz-/Drift-Report

Optional, in der Fallstudie ledger-casestudy: ./demo-gate-at-failure.sh — das Gate verwirft ein falsches Buchungsmodell reproduzierbar (grün → rot → grün).
